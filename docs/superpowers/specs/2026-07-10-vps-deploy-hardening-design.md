# VPS Deploy Hardening + First-Boot Admin

**Date:** 2026-07-10  
**Status:** Approved direction (awaiting implementation plan)  
**Goal:** Make Scuffed Crew installable on a single VPS for real testing without Discord OAuth, with a novice-friendly path that does not require hand-editing `.env`.

---

## Context

The app surface is largely complete (org site, admin, tournaments, wiki, forum, Nostr optional, etc.). Deploy is only partly wired: `compose.yml`, `Containerfile`, Caddy template, restic scripts exist, but gaps block a clean first install (first admin, uploads persistence, env docs, relay build, cookie/TLS story).

Discord OAuth is **out of scope for first install** (operator has no Discord account). Google OAuth remains optional later.

---

## Decisions

| Topic | Choice |
|-------|--------|
| Orchestration | **Podman Compose** (`compose.yml`) on one host |
| Not now | Kubernetes / k3s |
| Privilege | Rootful or rootless Podman both documented; Compose is the same |
| Public TLS | **Host Caddy** (`deploy/Caddyfile`) reverse-proxy to `127.0.0.1:3000` |
| Infra secrets (DB, encryption) | **A+B: install script auto-generates** into a secrets file; novice never types DB password |
| First admin | **First-boot account creation** in the browser only |
| Auth after setup | Local username + password (Argon2id hash in DB) |
| Discord / Google | Optional; leave empty unless configured later |
| Relay (strfry) | Optional Compose **profile** (`relay`); not required for core site |
| Quadlet | **Optional later step only** — not part of this implementation |
| Non-root DB user | Document; app may keep root credentials for v1 simplicity |

## 0. Install path (novice-friendly): auto secrets + install script

### Problem

SurrealDB root password must exist **before** the DB container starts (compose `command: ... --pass …`). The browser cannot invent that password first. Novices should not hand-edit `.env` or understand database credentials.

### Solution (A + B)

| Piece | Role |
|-------|------|
| **A — Auto-generate** | If secrets file is missing, generate strong random `SURREALDB_PASSWORD` and `ENCRYPTION_KEY` (and any other infra secrets needed to boot). |
| **B — Install script** | `scripts/install.sh` is the happy path: create secrets if needed, optional prompts (e.g. public domain / `REDIRECT_BASE_URL`), then `podman compose up --build -d`. |

### Secrets file

- Path: e.g. **`data/secrets.env`** (or project-root `.env` generated by install — prefer a dedicated path under `data/` that is gitignored).
- Permissions: **`0600`**, owner = deploying user.
- **Never commit.** Ensure `.gitignore` covers it.
- Compose loads env from this file (`env_file:` and/or documented `podman compose --env-file`).
- Contents at minimum:
  - `SURREALDB_PASSWORD=` (random, high entropy)
  - `SURREALDB_USER=root` (or fixed default)
  - `ENCRYPTION_KEY=` (base64 32 bytes, for Nostr field encryption when used)
  - `HOST_PORT=` (free host port chosen at install; compose maps `127.0.0.1:${HOST_PORT}:3000`)
  - `REDIRECT_BASE_URL=` (from install prompt or default `http://127.0.0.1:${HOST_PORT}`)
- Optional later keys (empty until configured): Discord/Google, Matrix, etc.

**Novice never chooses or sees the DB password in the UI.** Day-to-day they only know the **admin** password from first-boot. Document that infra secrets live in `data/secrets.env` if recovery tools (backup export, exec into Surreal) need them.

### Install script behavior (`scripts/install.sh`)

1. Check Podman / compose available; fail with clear message if not.
2. If secrets file missing:
   - Generate `SURREALDB_PASSWORD`, `ENCRYPTION_KEY`.
   - **Pick a free host port** (`HOST_PORT`): try a short list of preferred ports (e.g. 3000, 8080, 9090, 3100), then fall back to a random port in a high range (e.g. 9100–29999) until `ss`/`/dev/tcp` shows it free. Bind only via compose as `127.0.0.1:${HOST_PORT}:3000` (loopback; not public by default).
   - Prompt optional: public site URL (for `REDIRECT_BASE_URL` and Caddy notes); blank → `http://127.0.0.1:${HOST_PORT}` for local/tunnel test.
   - Write secrets file mode 600, including `HOST_PORT=…`.
3. If secrets already exist: **do not regenerate** passwords or re-roll `HOST_PORT` (stable across restarts). Operator can edit `HOST_PORT` manually if they must move.
4. Run `podman compose up --build -d` (default profile: no relay) with `--env-file` secrets.
5. Print next steps: chosen port, open URL → create admin account; optional Caddy `reverse_proxy localhost:${HOST_PORT}`; where secrets live.

**Port selection notes:** Checking free-then-bind has a tiny race; on compose bind failure, install may re-pick once or instruct the operator to re-run after freeing a port. Prefer **persisted** `HOST_PORT` over a new random port every boot.

Power users may still copy `.env.example` and set `HOST_PORT` manually; install script is the documented default.

### What is *not* set in install

- **Admin username/password** — only in the browser first-boot wizard.

## 1. First-boot admin setup

### Setup incomplete when

- `SURREALDB_URL` is set (production/real DB path), **and**
- No member with `org_role = 'admin'` exists.

In-memory dev mode (`SURREALDB_URL` unset) keeps existing seed + `/api/dev/login`; first-boot UI is for remote DB only.

### API

| Endpoint | Behavior |
|----------|----------|
| `GET /api/auth/setup-status` | `{ "needs_setup": bool, "local_login": bool }` — public, no auth |
| `POST /api/auth/setup` | Body: `{ username, password }`. Allowed **only** when `needs_setup`. Creates local user + admin member + session cookie. Rate-limited. |
| `POST /api/auth/local/login` | Body: `{ username, password }`. Session cookie on success. Rate-limited. Enabled when at least one local credential exists. |
| `GET /api/auth/providers` | `{ local, discord, google }` booleans for SPA login UI |

After any admin exists, `POST /api/auth/setup` returns **403** (or 410). Concurrent first setups: only one wins; loser gets a clear error.

### Schema

- Extend `user.provider` assert to include `'local'`.
- Store password material only for local users (dedicated fields on `user` or `local_credential` table): Argon2id hash + optional salt if not embedded in hash string. Never store plaintext.
- No open registration beyond one-time setup. New members still come from applications / admin flows.

### UI

- On app load, call `setup-status`.
- If `needs_setup`: full-page **Create admin account** (username, password, confirm). Min password length **12**.
- Else: normal site; login offers local form when `local_login` / providers.local is true; Discord/Google buttons only when configured.

### Password recovery (operator-only)

**Admin password:** no email recovery. Document:

1. Optional emergency: `BOOTSTRAP_ADMIN_RESET=1` + `BOOTSTRAP_ADMIN_PASSWORD=...` + username (one restart resets that local user’s hash), **or**
2. Small `scripts/reset-local-admin.sh` via `podman exec` / Surreal.

Day-to-day: password manager stores the password chosen at first boot.

**Database password:** not a human login. Recover from `data/secrets.env` (or rotate by advanced procedure documented later; not first-install scope).

### Security

- Rate-limit setup + local login (same class as OAuth governor).
- Release builds use Secure cookies → HTTPS via Caddy required for real logins on a public host.
- Setup must not remain open after first admin.
- Secrets file never world-readable; not logged by install script.

## 2. Compose / runtime

### Services (default)

1. **surrealdb** — SurrealDB v3, named volume, internal network, healthcheck.
2. **site-server** — build `Containerfile`, depends on healthy DB, publish `127.0.0.1:3000:3000`, internal + external networks.

### Optional profile `relay`

- **strfry** (+ relay-policy) only with `podman compose --profile relay up`.
- Fix `relay/Containerfile` so workspace cargo build works (copy required crates / workspace files completely enough for `-p relay-policy`).

### Volumes

| Volume | Mount |
|--------|--------|
| `surrealdb-data` | DB rocksdb path (existing) |
| `uploads-data` | `/app/data/uploads` on site-server; `UPLOAD_DIR=/app/data/uploads` |
| `strfry-data` | only with relay profile |

### Env

- Happy path: compose reads **generated secrets file** from install.
- Keep **`.env.example`** as a reference for power users / documentation of every variable (not the novice primary path).
- Compose passthrough for Surreal, redirect URL, encryption, optional OAuth/Matrix, emergency admin reset vars.

### Bootstrap DB script

Keep `scripts/bootstrap-db.*` for optional scoped user. Document that with compose, Surreal is **not** on host localhost — use `podman exec` into the DB container. App may continue using root credentials for v1.

## 3. Reverse proxy

- Host **Caddy** (or equivalent) using `deploy/Caddyfile` (domain replace).
- Proxy to `localhost:3000`.
- `REDIRECT_BASE_URL` set by install script when operator provides domain.
- Not containerized in this pass (keeps cert storage simple).

## Success criteria

1. **Novice path:** run `./scripts/install.sh` without hand-editing `.env`. Install **auto-generates** `SURREALDB_PASSWORD`, `ENCRYPTION_KEY`, and a free **`HOST_PORT`** into a mode-`600` secrets file (idempotent if file already exists — port and secrets stay stable).
2. Optional install prompt for public URL only; **no** prompt for database password, admin password, or host port during install (port is auto-picked).
3. `podman compose up` (via install) brings up DB + site without Discord (or any OAuth).
4. First visit shows **Create admin account**; operator chooses **username + password there**. That admin password is hashed into the DB; it is never taken from env for the normal path.
5. After setup, session works on HTTPS (or documented HTTP-only local caveat); later visits use **local login** with the first-boot password.
6. Admin panel usable; data (DB + uploads) survives container recreate.
7. `docs/deploy.md` documents happy path, secrets file location, Caddy, backups, recovery, and Quadlet only as an optional later step.
8. Relay is opt-in; core path does not depend on it.
9. Emergency admin password reset (env or script) is recovery-only, not required for first install.
