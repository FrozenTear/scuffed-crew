# USER ESCALATION — DR1-NOSTR-001 Nostr challenge deterministic-key fallback

**RESOLVED SEVERITY: HIGH** (grok filed CRIT; independent replay verifier settled
it at HIGH — not CRIT). Real bug, real fix, but **not remotely exploitable by a
passive attacker** — the deciding barrier is capture-feasibility (see below).

**Why HIGH not CRIT:** the login-replay path IS real and unblocked (challenge
verification is stateless — no nonce/consumed-challenge store, and nostr 0.44.2
`Event::verify()` checks only id+signature, NO `created_at` freshness, so a
victim-signed event verifies forever). BUT the kind-22242 login event is POSTed
**directly** to `/api/auth/nostr/verify` and is **never published to a relay**
(login.rs:543-568) — capturing one needs TLS MITM / a malicious NIP-07 extension
/ server-side log compromise, not passive observation. That caps it at HIGH.

**Still worth doing on your instance** (deterministic public key is bad hygiene
regardless): set `NOSTR_CHALLENGE_SECRET` to a random 32-byte value and restart.

## What is definitely true

- `NOSTR_CHALLENGE_SECRET` is **not generated by `install.sh`** (it generates
  `ENCRYPTION_KEY` but not this) and `compose.yml:81` defaults it to empty.
- When it is missing/empty, both server binaries derive the challenge-token MAC
  key from the **hard-coded public string** `scuffed-crew-dev-nostr-challenge-key`
  (`crates/site-server/src/main.rs:476-490`, `crates/server/src/main.rs:155-163`).
- Production does **not** fail closed — it only suppresses the dev warning
  (`is_dev` branch). So a standard `install.sh` deploy runs with a
  publicly-known MAC key. **This includes your live instance if you have not
  manually set the secret.**

## Where grok's CRIT is bounded (claude cross-check)

The direct "forge a token for any member_id → take over that account" path is
**blocked**: `nostr_verify` (`routes/nostr.rs:249`) requires an authenticated
`caller: OrgMember`, rejects the request unless `token_member_id ==
caller.member.id` (nostr.rs:266), and writes the pubkey to `caller.member.id`
regardless of the token. The login flow (`routes/auth.rs:756`) derives identity
from the **verified event signature's pubkey** (auth.rs:779-782), not the MAC.
So MAC forgery alone does not impersonate.

## The residual risk that keeps it serious (claude, not in grok's write-up)

In the login flow the MAC+TTL is the **replay protection** for a captured
victim-signed kind-22242 event. With a known key an attacker can re-mint a
fresh-expiry token wrapping an **old captured** `scuffedclan-login:` challenge
the victim already signed, then submit it → session as the victim, no victim
key needed. This is CRIT **iff** those ephemeral events are capturable (relay
broadcast / MITM). A dedicated verifier is checking feasibility now.

## Fix (same at CRIT/HIGH/MED — safe to land overnight under dual-agree)

1. **Fail closed:** in both binaries, if not dev and `NOSTR_CHALLENGE_SECRET`
   is missing/empty, refuse to boot (mirror the existing `ENCRYPTION_KEY` /
   `PRODUCTION` refuse-to-boot pattern) — do not mount nostr routes with the
   fallback key.
2. **Provision it:** `install.sh` generates `NOSTR_CHALLENGE_SECRET` like it
   does `ENCRYPTION_KEY`; `update.sh` appends it to existing `secrets.env`.
3. **Operational (your instance):** set `NOSTR_CHALLENGE_SECRET` to a random
   32+ byte value and restart — this closes it immediately regardless of the
   code fix. Any existing challenge tokens become invalid (fine; they're short-TTL).
