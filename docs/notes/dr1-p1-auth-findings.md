# DR-1 P1 — AUTH lane findings

Reviewer: claude (Opus) · Date: 2026-07-19 · Scope: `crates/auth/**` + `crates/site-server/src/extractors.rs`
Usage-side files (`routes/auth.rs`, `lib.rs`, `server/main.rs`, `db/queries/sessions.rs`) traced only where the
brief explicitly required assessing an AUTH property (rate-limit posture, timing oracle, session revocation on
credential change, secret-in-logs). Findings on those files are labelled (boundary).

Legend: CONFIRMED = code traced end-to-end. SUSPECTED = needs more digging / exploitability conditional.

---

## Findings

DR1-AUTH-001 | MED | crates/site-server/src/lib.rs:51-76 (+SmartIpKeyExtractor) | CONFIRMED (boundary) | The sole brute-force / credential-stuffing control is a `tower_governor` per-IP limiter whose `SmartIpKeyExtractor` derives the key from client-controllable forwarded headers (X-Forwarded-For / X-Real-IP / Forwarded) before falling back to the peer socket. | Attacker sends `POST /api/auth/local/login` with a rotating `X-Forwarded-For: <random-ip>` header on every request. Each distinct header value gets its own token bucket, so the 5-burst/1-per-2s cap never applies → unlimited password / credential-stuffing attempts against local login (and unlimited hammering of nostr/verify, setup, register). install.sh only *suggests* fronting with Caddy at 127.0.0.1 (scripts/install.sh:163) and does not enforce it; a direct `-p 0.0.0.0:PORT` podman publish, or a Caddy that appends rather than replaces XFF, exposes this fully. 12-char min + Argon2 blunts naive guessing but not credential stuffing with breached passwords. | Only honor forwarded IPs from a configured trusted-proxy allowlist (or use the peer socket when not behind a known proxy); document that the container port must bind 127.0.0.1 and sit behind a header-sanitizing proxy. Consider a separate, stricter per-account failure counter on local_login independent of source IP.

DR1-AUTH-002 | MED | crates/site-server/src/routes/auth.rs:654-676 + crates/auth/src/password.rs:33 | CONFIRMED (boundary) | `local_login` runs Argon2 `verify_password` only when the username row exists; the no-user branch (`Ok(None) => return invalid()`) returns immediately with no dummy hash → username-enumeration timing oracle. | Attacker POSTs login with a candidate username + arbitrary password. Existing username → server spends Argon2 verify time (tens of ms); non-existent username → returns in sub-ms. Response-time delta reveals which usernames are registered, enabling targeted attacks. Amplified by DR1-AUTH-001 (rate-limit bypass lets the attacker collect clean timing samples). | On the no-user path, verify the supplied password against a fixed precomputed Argon2 dummy hash (with the same params) and discard the result, so both paths spend equivalent time before returning the generic `invalid` error.

DR1-AUTH-003 | MED | crates/server/src/main.rs:113-138 | CONFIRMED (boundary) | `BOOTSTRAP_ADMIN_RESET` rewrites a local admin's password hash via `set_local_password_hash` but never calls `delete_sessions_for_user` — unlike every other credential/mod mutation in the codebase (admin reset route members.rs:837, ban moderation.rs:203, deactivate members.rs:344), which all revoke sessions. | Emergency admin-recovery is exactly the "account compromised / attacker holds a live session" scenario. Operator sets BOOTSTRAP_ADMIN_RESET=1 + new password and restarts. Password changes, but the attacker's existing DB session row survives (sessions persist across process restart), so the attacker keeps admin access despite the password reset. | After a successful `set_local_password_hash` in the bootstrap path, call `delete_sessions_for_user(&user.id)` to enforce the same credential-change-revokes-sessions invariant used elsewhere.

DR1-AUTH-004 | MED | crates/auth/src/crypto.rs:213-215 (vs db require_oauth_encryption) | CONFIRMED gate logic / SUSPECTED exploitability | The empty-AAD legacy decrypt fallback (`allow_legacy_empty_aad`) is disabled only by `CRYPTO_STRICT_AAD=1` OR `PRODUCTION` truthy. It is NOT keyed off "remote SurrealDB / encryption-required", whereas `require_oauth_encryption` (db/queries/users.rs:108) forces encryption whenever PRODUCTION **or** remote DB. Result: a real remote-DB deploy that omits `PRODUCTION=1` (env dropped from compose, manual `cargo run`, non-install.sh operator) still mandates encryption but silently keeps the AAD downgrade fallback on. | On such a deploy, any ciphertext that fails AAD-bound decryption is retried with empty AAD (crypto.rs:313-326). New app writes always use `aad::*`, so on a fresh DB this is inert; but if legacy pre-AAD blobs exist (post-upgrade), their domain-separation binding (e.g. `nostr_secret_key(pubkey)`, `oauth_provider_id(user,provider)`) is not enforced — an actor able to relocate such a blob across records (confused-deputy / DB write) bypasses the context binding. Defense-in-depth degradation, not a standalone break; install.sh:81 does set PRODUCTION=1 so the blessed path is safe. | Enable strict AAD whenever encryption is required (mirror `require_oauth_encryption`'s remote-DB condition), not only when PRODUCTION is truthy; treat empty-AAD fallback as opt-in via an explicit legacy-migration flag.

DR1-AUTH-005 | LOW | crates/auth/src/crypto.rs:376-380 | CONFIRMED | `hash_provider_id` is an unsalted, keyless, deterministic BLAKE3 of `provider:provider_id`, used as the DB lookup key for OAuth accounts. | If the `provider_id_hash` column leaks (backup, read-only injection elsewhere), an attacker can brute-force it offline: Discord/Google IDs are numeric snowflakes with predictable structure/entropy and BLAKE3 is very fast, so many real account IDs are recoverable via dictionary/range enumeration, linking site accounts to Discord/Google identities. | Accepted trade-off for deterministic lookup, but consider a keyed hash (BLAKE3 keyed with a server secret / HMAC) so a column leak alone is not offline-enumerable.

DR1-AUTH-006 | LOW | crates/auth/src/server/oauth.rs:122-145 + google.rs:33 | CONFIRMED | OAuth uses cookie-state CSRF only, no PKCE; Google login hits `/oauth2/v2/userinfo` with the access token and never verifies an OIDC `id_token`/nonce. | For a confidential server-side client with a client secret this is acceptable, but PKCE + id_token/nonce validation would harden against authorization-code interception and token-substitution/confused-deputy edge cases. Low residual risk given the secret-bearing exchange. | Add PKCE (S256) to the auth-code flow and validate Google's id_token signature+nonce+aud instead of the plain userinfo endpoint.

DR1-AUTH-007 | LOW | crates/auth/src/server/oauth.rs:102-119 | CONFIRMED | `create_client` is called per-request inside `get_auth_url`/`exchange_code` and `expect`/`panic!`s on a malformed auth/token/redirect URL. | If an operator misconfigures `REDIRECT_BASE_URL` (or a provider URL) to something unparseable, every login/callback request panics. Axum isolates the panic to a 500 per request, but it is a self-inflicted availability footgun and noisy. | Validate provider config once at startup (fail fast), or return `AuthError`/500 gracefully instead of panicking in a request handler.

DR1-AUTH-008 | LOW | crates/auth/src/types.rs:97-110 + server/session.rs:25 | CONFIRMED | Session lifetime is a fixed 168h (1 week) absolute expiry with no idle/inactivity timeout and no rotation on privilege change. | A stolen session token is valid for up to a week regardless of inactivity. Mitigated by: token is HttpOnly + Secure (release) + hashed at rest, MAX_SESSIONS_PER_USER=10 cap, and per-request re-read of role/active/suspended from DB (extractors.rs:42-84) so privilege downgrades take effect immediately without needing revocation. | Consider a sliding idle timeout and/or shorter absolute lifetime for privileged accounts; low priority.

DR1-AUTH-009 | LOW | crates/site-server/src/extractors.rs:90-102 | CONFIRMED | `OptionalOrgMember` maps *every* `OrgMember` error (including 500 DB errors) to `None`, so on a transient DB fault an authenticated member silently degrades to the anonymous view, and a suspended/inactive member is indistinguishable from anon. | Not a privilege escalation (anon ≤ member), but it masks infrastructure failures on optional routes and can silently show stale/anon content during DB blips. | Distinguish "no auth" from "auth infra error"; at minimum log the swallowed error. Confirm no optional route grants anonymous a capability it would deny a suspended member.

DR1-AUTH-010 | NIT | crates/auth/src/server/matrix.rs:37-45 | CONFIRMED | `MatrixProvider::config` returns `https://example.com/{authorize,token,userinfo}` TODO stubs. | Verified NOT reachable: the login/callback dispatch (routes/auth.rs:56-98) only matches `"discord"`/`"google"`; Matrix is never registered, so the example.com endpoints cannot be hit. Risk only materializes if someone wires Matrix without replacing the stubs. | Gate the stub behind `unimplemented!()`/feature flag, or add a test asserting no provider config points at example.com, so a future wiring can't silently ship pointing at example.com.

DR1-AUTH-011 | LOW | crates/site-server/src/lib.rs:35-46 + auth/src/server/session.rs:20-26 | CONFIRMED | API mutations (POST/PUT/PATCH/DELETE) carry no per-request CSRF token; cross-site protection rests entirely on `SameSite=Lax` session cookies + the `ALLOWED_ORIGINS` CORS allowlist. | Acceptable under the current design (allowed_origins defaults to a single origin and fails closed if misconfigured; no state-changing GET routes exist in prod — dev_login GET is dev-gated). Becomes exploitable if a state-changing GET route is ever added (Lax sends the cookie on top-level GET navigations) or if SameSite is relaxed. | Keep the invariant "no state-changing GET"; consider a defense-in-depth double-submit CSRF token for mutations. Document the ALLOWED_ORIGINS-strictness dependency.

## Refuted recon claims (traced and disproved — record so P2 doesn't re-raise)

- **Rate-limiting is NOT absent.** Recon (dr1-p0-targets-claude.md:58) said "Brute-force/rate-limit posture: NONE found". A `GovernorLayer` (per-IP, 5-burst + 1/2s) wraps the auth_routes group covering login, callback, setup, local/login, local/register, nostr/challenge, nostr/verify (lib.rs:51-76). ConnectInfo is wired in both binaries (main.rs). The real weakness is DR1-AUTH-001 (the extractor trusts spoofable forwarded headers), not absence.

- **Secure cookie is NOT false in release builds.** Recon (targets:28) said Secure "can be FALSE in a release build that isn't production and lacks SECURE_COOKIES". `is_secure = SECURE_COOKIES set || is_production_env() || !cfg!(debug_assertions)` (session.rs:16-19, 96). In any `--release` build `debug_assertions` is off, so `!cfg!(debug_assertions)` is `true` and Secure is unconditionally set. The only Secure=false path is a *debug* build without SECURE_COOKIES/PRODUCTION (i.e. local dev). Production containers build release → Secure always on.

## Confirmed-clean (in scope, no finding)

- Session storage: raw tokens are 32B OsRng, stored only as BLAKE3 hashes; get/delete/logout all hash before querying; revocation is bulk-by-user for ban/deactivate/reset (sessions.rs). Logout deletes the DB row (not just the cookie). No secret material logged (tracing logs usernames/user_ids/counts only; keys are Zeroized).
- NIP-49 (nip49.rs): scrypt log_n bounded 16..=20 (DoS guard, tested), zeroize on all paths, version/length checks. Clean.
- AES-GCM core: 12B OsRng nonce per op, AAD bound, Zeroizing plaintext, key len checked + zeroized, previous-version collision rejected, tamper/wrong-key/wrong-AAD tests present. Clean apart from DR1-AUTH-004's gate scoping.
- CSRF state validation (session.rs:45-84): CsrfToken::new_random, cookie-vs-param compare, all None/mismatch paths → 400. Cookie is the binding; validated before code exchange (auth.rs:100-104). Acceptable (see DR1-AUTH-006 for PKCE defense-in-depth).
- Extractor chain (extractors.rs): fail-closed on DB error (500), fresh per-request read of member is_active/suspended/role; OrgMember→OfficerUser→AdminUser tiering correct; DaemonUser re-checks active + suspended/banned.
- CORS: explicit origin allowlist (not wildcard) with allow_credentials; defaults to a single origin and fails closed.
