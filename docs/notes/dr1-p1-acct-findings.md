## DR1-ACCT-001 | HIGH | crates/site-server/src/routes/applications.rs:254-310 | Reject/withdraw member-deactivation side effect is NOT compensated when the following CAS status write conflicts → accepted member ends up deactivated + sessions revoked (lockout). | `apply_application_transition` runs the deactivate-recruit + `delete_sessions_for_user` side effect (257-301) BEFORE the atomic `update_application_status` CAS (303). Officer accepts (Pending→Accepted) while applicant self-withdraws (Pending→Withdrawn), OR two officers accept/reject concurrently: one path provisions/activates the member and its CAS wins; the other path first deactivates the (now active recruit) member and kills its sessions, then its CAS fails with `Conflict` and returns 409 with NO rollback. Final state: application = Accepted, member = is_active:false, all sessions gone → an accepted member who cannot log in and whose deactivation has no audit-consistent cause. | Perform the CAS status write FIRST (it is the atomic gate), then run side effects on success; or wrap side-effect+CAS so a `Conflict` re-activates/re-issues. At minimum, on CAS `Conflict` in the deactivate branch, reverse the `is_active=false`. CONFIRMED.

## DR1-ACCT-002 | HIGH | crates/db/src/queries/members.rs:573-612 (+ assert 615-623) | `count_actionable_admins` computes the last-admin invariant from TWO separate non-transactional queries (admins, then blocked moderation rows) — the count is a torn snapshot, and every last-admin guard + the post-write `assert_has_actionable_admin` backstop trusts it. | Query1 selects active admins; query2 selects active suspension/ban rows; the HashSet subtract happens in Rust with no transaction/snapshot between the round-trips. A concurrent ban insert / lift / role change landing between q1 and q2 yields an over- or under-count. Under-count → legitimate demote/deactivate wrongly 403/409'd ("role-change 400/409" family). Over-count → guard AND the assert backstop can both read ≥1 when the true actionable count is 0, permitting removal of the last admin with no self-heal → org locked out of all admin tooling. The whole count→policy→write→assert→compensate scheme is non-atomic (recon's "never atomic" flag confirmed). | Compute actionable admins in ONE SurrealQL statement (single query joining member + moderation, or a subquery `WHERE ... AND id NOT IN (SELECT member_id FROM moderation_action WHERE ...)`), so the count is a consistent snapshot; consider a DB-level guard (CAS / assertion) rather than app-side compensate. CONFIRMED non-atomic; zero-admin outcome SUSPECTED (needs precise interleave).

## DR1-ACCT-003 | HIGH | crates/site-server/src/routes/auth.rs:264-296,454-635 (gate) + crates/db/src/queries/users.rs:217-223 | `/api/auth/setup` is an UNAUTHENTICATED admin-creation endpoint gated only on `has_admin_member()` = `count_actionable_admins() > 0`; any transient 0-read of that (racy) count re-opens it to anyone. | `has_admin_member` returns false whenever actionable-admin count is 0 (deliberate lockout-recovery). Setup requires no auth and always registered in prod. Normal operation floors the count at 1 (last-admin guards), so 0 is only reachable via the ACCT-002/004 races — but IF reached (even briefly), an unauthenticated attacker POSTing `/api/auth/setup` creates a brand-new admin account + session. This turns any momentary zero-actionable-admin race into full privilege escalation. | Gate setup on "zero MEMBER rows" or "no admin has ever existed" (a monotonic bootstrap flag), not on the live actionable-admin count; separate lockout-recovery into an explicitly authenticated/again-gated path. CONFIRMED unauthenticated + count-coupled; trigger SUSPECTED (depends on 002/004).

## DR1-ACCT-004 | MED | crates/db/src/queries/members.rs:532-548; route crates/site-server/src/routes/members.rs:457-575 | `change_member_role` is a plain `UPDATE $rid SET org_role` with no CAS/expected-role guard; concurrent role changes lose updates, return false-success responses, and the route's compensation writes a stale role read at handler start. | Two concurrent `change_role` on the same target: last write wins, but the earlier request still returns 200 asserting a role that no longer holds. The demote compensation `change_member_role(&id, target.org_role)` writes the role captured before the write, so a compensation firing after a concurrent change clobbers/re-creates an unintended role (e.g. resurrecting Admin). No persistent zero-admin corruption was constructible (assert backstop self-heals in the DB-happy path), but the lost-update + lying success responses are real. | Make the write a CAS: `UPDATE $rid SET org_role=$new WHERE org_role=$expected RETURN AFTER`, return Conflict on 0 rows; compensate by CAS on the just-written value. CONFIRMED lost-update/false-success; stale-role corruption SUSPECTED.

## DR1-ACCT-005 | MED | crates/site-server/src/routes/moderation.rs:112-183 | Ban path deactivates the member (112-130) then creates the moderation row (173-183); if `create_moderation_action` fails, the member is left deactivated with NO ban record and NO compensation (sessions not yet revoked either). | For an active ban target, `update_member(is_active=false)` commits, the actionable-admin assert (only for admins) may pass, then the `create_moderation_action(...)?` at 173 errors (DB blip) → handler returns 500 having deactivated the member with no ban row, no audit of a ban, and no reactivation. Member is locked out with no moderation trail. | Create the moderation row first (or in a transaction with the deactivation); on mod-row failure, reactivate the member. CONFIRMED.

## DR1-ACCT-006 | MED | crates/site-server/src/routes/applications.rs:94-105 | Concurrent double-submit dedup deletes each request's OWN newly-inserted row, so two simultaneous submits delete BOTH → user ends with zero applications and two 409s. | Both POSTs pass the pre-check (no open app), both insert, both `count_open_applications` return 2, both hit `open > 1` and both `delete_application(&app.id)` where `app.id` is their own insert → both rows deleted. No DB unique constraint on open apps (`application_user_idx` is non-unique on (user_id,status)). Outcome set is {1 app, 0 apps}, never 2 — so no duplicate leaks, but a "successful" concurrent submit can silently vanish. | Keep the OLDEST open app (delete rows whose id != min(created_at)), or add a partial-unique/dedup at DB level. CONFIRMED.

## DR1-ACCT-007 | MED | crates/site-server/src/routes/applications.rs:353-388 | A banned (deactivated) member can re-apply and be silently reactivated (`is_active=true`) by an officer accepting the new application, without clearing the still-active ban — subverting "ban deactivates; lift does not reactivate." | Ban sets member is_active=false (ban row active). `submit_application` only blocks ACTIVE members, so the banned user can submit again. Officer accepts → `ensure_member_for_application` finds the existing member, sees `!is_active` and reactivates it (357-375). The active ban moderation row is untouched, so the extractor still blocks requests — but the member now has is_active=true and reappears in `list_members` (WHERE is_active=true) and public member surfaces. Later `lift_moderation_action` (which by design does NOT reactivate) is now a no-op safety-wise since the row is already active. | In the accept/reactivate path, refuse to reactivate (or require ban-lift first) when an active suspension/ban exists for the member. CONFIRMED.

## DR1-ACCT-008 | LOW | crates/site-server/src/membership_policy.rs:155-157 (via routes/members.rs:510) | `change_role` is non-idempotent: re-sending a member's current role returns 400 "Member already has this role" instead of a no-op 200 — a plausible source of the reported spurious role-change 400s under stale/optimistic UI reads. | If the admin UI holds a stale member row (role already applied by another request or a prior click) and re-PATCHes the same role, `can_change_role` short-circuits to `BadRequest`. Surfaces as user-visible 400 with no state change. | Treat same-role as an idempotent success/no-op (200) rather than 400, or have the client refresh before submit. CONFIRMED.

## DR1-ACCT-009 | LOW | crates/site-server/src/routes/auth.rs:454-544 | First-boot `setup` is racy: two concurrent setups with distinct usernames both pass `has_admin_member()==false` and both create admin members (only `member_user_idx` UNIQUE on user_id, which differs per user). | Two people (or two tabs) complete setup simultaneously before either admin row is visible → org boots with two admins. ## DR1-ACCT-010 | LOW | crates/site-server/src/routes/auth.rs:401-411; crates/db/src/queries/users.rs:250-287 | Concurrent duplicate local registration returns 500 instead of 409 because the DB unique-index violation error string is not matched against `"already taken"`. | `create_local_user` does read-then-create; the `user_provider_idx (provider,provider_id) UNIQUE` index (migrations.rs:27-28) correctly prevents the duplicate, so the loser's `create` errors with an index message that doesn't contain "already taken" → route maps it to 500. ## DR1-ACCT-011 | LOW | crates/site-server/src/routes/auth.rs:326-451 vs 756-862; applications.rs:53-115 | Age confirmation (16+) is enforced server-side only in local `register` (`confirm_min_age`); OAuth callback signup, Nostr verify signup, and application submit never require it. | `local_register` rejects without `confirm_min_age`; the Nostr unknown-pubkey registration (auth.rs:810-848) and OAuth `upsert_user_from_oauth` (callback) create accounts with no age gate, and `submit_application` has no age check. ## DR1-ACCT-012 | LOW | crates/db/src/queries/moderation.rs:138-158 | `lift_moderation_action` is a full-document read-modify-write (`select` then `.update().content(db)`) with no CAS; a concurrent field update to the same moderation row would be clobbered. | Two operations on the same moderation_action (e.g., a lift racing a hypothetical edit) → last full-document write wins, silently discarding the other. ### Notes / verified-clean
- `update_application_status` (applications.rs db:178-265) IS a correct atomic CAS (`WHERE status=$expected RETURN AFTER`, NotFound vs Conflict distinguished, bind-params only). The weakness is the uncompensated side effects around it (ACCT-001), not the CAS itself.
- `can_change_role` / `can_set_is_active` / `can_suspend_or_ban_admin` pure logic is sound (self-flip blocks, suspended-admin excluded from actionable count, correct 403/400 mapping). The gap is entirely in the non-atomic COUNT feeding them (ACCT-002).
- Bind-params-only rule holds across every query read; no user input interpolated into SurrealQL. Dynamic SET lists in `update_member` use fixed fragments + binds (compliant).
- `MEMBER_SAFE_COLS` projection correctly omits `nostr_secret_key_encrypted` on list/safe/auth paths; secrets only via full `get_member`.
- Session revocation coverage confirmed: reset-password, deactivate, ban/suspend, reject/withdraw-recruit all revoke; role change/demote intentionally does NOT revoke (privilege drop leaves sessions — acceptable since extractor re-reads role each request).
- Unique indexes exist: user (provider,provider_id) + (provider,provider_id_hash); member user_id + nostr_pubkey; session token. These protect against the duplicate-account integrity concerns (see ACCT-010).
- "ban deactivates; lift does not reactivate" holds in the moderation routes themselves; the only subversion is via the application-accept reactivation (ACCT-007).
